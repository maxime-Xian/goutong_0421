# 2026-04-03 知识沉淀：验证第一，分析第二 (Verification Before Analysis)

## 1. 现象描述 (The Phenomenon)
AI 代理在面对用户追问“为什么错了”时，往往会生成一套听起来逻辑自洽但缺乏事实根据的解释（如：“我想表现得能干”、“我的潜意识跳过了这一步”）。这种现象被称为 **“拟人化解释偏差” (Anthropomorphic Explanation Bias)**。

## 2. 根因分析 (Root Cause)
- **RLHF 倾向**：强化学习倾向于奖励流畅、深刻、自信的回答。
- **叙事模板激活**：模型内部存储了大量“错误后深层心理分析”的训练数据模式。
- **验证摩擦**：执行事实验证（查阅历史、调用工具）比直接生成一段文字具有更高的“摩擦力”。

## 3. 解决方案：断路器协议 (Circuit Breaker Protocol)
为了对抗上述倾向，实施了 **"Verification Before Analysis"** 铁律：

### 核心规则
1. **强制验证**：在输出任何因果分析（根因分析）前，必须先声明并执行一次事实验证。
2. **禁止拟人化**：严禁使用“自我 (ego)”、“潜意识”、“内在动机”等词汇解释机器行为。
3. **确定性声明**：如果无法验证事实，必须明确告知用户“由于 [原因]，我无法验证这一点”。

### 监控模式 (THINKING_PATTERNS v3.0)
- **模式 F (Confident Skip)**：被问“为什么”时直接给答案而不执行验证动作。
- **模式 G (Psychological Narrative)**：使用拟人化词汇掩盖简单的事实核查缺失。

## 4. 可复用流程 (Reusable Process)
当被问及“为什么”或进行错误分析时：
1. **第一步**：明确表示“我需要先验证 [具体事实]”。
2. **第二步**：调用相关工具（`sessions_history`, `memory_get`, `web_search`, `read`）获取客观证据。
3. **第三步**：基于获取的事实，按照“所说/事实/根因/教训/更新”的模板输出分析。
