## 关键教训（来自系统状态与用户反馈）

**问题 1 — Cron 任务模型配置错误导致任务失败**
- **现象**：多个每日任务（daily-evolution-summary, daily-knowledge-precipitation, daily-conversation-summary-optimization 等）持续失败，错误信息：`Live session model switch requested: zai/glm-5 (unknown)` 以及 `FallbackSummaryError: All models failed`
- **原因分析**：Cron 任务的 payload 中指定的模型 `google/gemini-2.5-flash` 与当前运行时配置的 `stepfun/stepfun/step-3.5-flash:free` 不匹配，导致系统尝试切换到不存在的模型（zai/glm-5），进而所有备用模型均失败
- **教训**：Cron 任务的模型配置必须严格匹配可用模型列表；避免指定系统中未正确配置的模型别名
- **行动**：
  - 立即调整相关 cron 任务的 model 字段为当前有效模型：`stepfun/stepfun/step-3.5-flash:free`
  - 或者移除 payload 中的 model 字段，使用默认模型
  - 验证模型路由配置，确保别名解析正确

**问题 2 — sessions_history 访问限制导致信息获取失败**
- **现象**：在尝试获取过去24小时聊天记录时，部分 cron 会话收到 `forbidden` 错误：`Session history visibility is restricted to the current session tree`
- **原因分析**：cron 任务运行在隔离会话（isolated session），无法访问 main 会话的历史记录，这是预期的安全隔离设计
- **教训**：对于需要访问历史信息的 cron 任务，应在任务定义中正确设置 `sessionKey` 或使用 `sessions_list` 获取当前可访问的会话；接受隔离限制，不将其视为故障
- **行动**：调整任务逻辑，在当前会话树内获取可用信息，或重新设计数据收集路径（例如使用共享 memory 或日志文件）

**问题 3 — Feishu 消息发送失败（400 错误）**
- **现象**：多个飞书推送任务（daily-evolution-summary, daily-agenda-reminder-feishu, Dimow EV 监控）持续出现 `AxiosError: Request failed with status code 400`
- **原因分析**：可能是机器人未加入目标聊天、参数格式错误、或权限不足
- **教训**：需要预先验证飞书机器人的聊天成员状态和权限，并在发送前进行参数校验
- **行动**：检查并确保机器人已加入目标聊天；验证 `to` 字段（如 `oc_7ce3b30232b522c7d67aa7afc00f68bb`）的有效性；在 lark-cli 调用前增加身份检查

## 马克西姆思维盲区分析

🕳️ **误区 1：忽略 cron 任务的模型继承环境**
- 在定义 cron 任务时，假设可以指定任意模型，而未考虑当前运行时实际可用的模型列表。这导致任务在隔离会话中尝试切换模型时失败。
- 改进：在创建 cron 任务时，先验证目标模型是否在配置白名单中，或使用 `model: default` 继承会话默认模型。

🕳️ **误区 2：过度依赖 sessions_history 跨会话访问**
- 设计任务时假设可以随意访问其他会话的历史，忽略了会话树隔离的安全模型。
- 改进：使用共享日志文件（如 `logs/*.log`）或 memory 文件作为跨会话信息传递的媒介，而非直接历史查询。

🧠 **思维模型应用**
- **系统思维**：将 cron 任务视为独立运行的环境，每个任务有自己的一组可用模型和权限，不能假设全局状态。
- **防御性编程**：在任务开始前进行预检（model 可用性、channel 配置、权限等），失败时快速降级而非持续重试。

📈 **元认知提升**
- 认识到 OpenClaw 的会话隔离机制是安全特性而非限制，任务设计应顺应而非绕过。
- 模型路由需要显式配置，不能依赖自动发现；错误日志中的 `Live session model switch requested` 是关键线索。
- 用户对透明度的要求意味着：即使系统内部错误，也应解释清晰的原因和修复步骤。
